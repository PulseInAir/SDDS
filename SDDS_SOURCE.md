## 2. Product identity

**Product name:** SDDS ITR Practice Management System  
**Owner/use:** Private internal system for SDDS.  
**Product type:** Cloud-hosted internal web application; **not** a public SaaS and **not** a marketplace for other tax consultants.  
**Access:** Secure login from any device/location with internet access.  
**Primary deployment:** Vercel through GitHub.  
**Primary backend:** Supabase.

### Core business outcome

Create one reliable source of truth for every ITR client and every assessment year, covering:

- client identity and contact information;
- secure ITR portal credentials;
- document collection and document history;
- filing workflow and deadlines;
- original, revised, updated, rectification, and related filing records;
- e-verification and ITR-V tracking;
- intimation, refund, notice, and rectification tracking;
- invoices, payments, outstanding balances, and revenue;
- notes, communication history, follow-up, and future-year continuity.

The system must reduce missed filings, missing documents, forgotten payments, unresolved intimations/notices, and dependence on scattered files or memory.

---

## 3. Non-negotiable product rules

1. One client profile is the single source of truth for that client.
2. Data is organised by **client + assessment year**, while preserving full history across years.
3. The current assessment year is selected by default; users can switch years manually.
4. Filing a return must not overwrite prior filing records. Revised or subsequent filings require separate linked records.
5. PAN is stored and displayed in uppercase.
6. The ITR portal password is mandatory for applicable clients, encrypted at rest, hidden by default, and revealed only through an intentional per-record action.
7. Uploaded documents are private and cannot be exposed through permanent public URLs.
8. A filed case is not automatically “complete” until required post-filing steps are finished, including e-verification/ITR-V and other applicable closure checks.
9. A client is automatically placed into the next-year follow-up cycle after filing/completion unless manually excluded. Excluded clients remain recoverable and can be re-enabled.
10. Device loss must not cause business-data loss.
11. The system must support automatic daily backup and one-click export of clients, filings, invoices, payments, refunds, and the complete database in practical formats.

### Environment-variable rules

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`

Use the service-role key only in server-only code. Never expose it through `NEXT_PUBLIC_*`, client bundles, logs, or browser requests.

### Storage decision

Google Drive is not the SDDS document-storage layer for the approved MVP architecture. Use Supabase Storage.

- Private bucket: `sdds-documents`
- Recommended path:
  `clients/{clientId}/{assessmentYear}/{documentType}/{timestamp}-{safeFileName}`
- Store file metadata in the database.
- Generate short-lived signed URLs through authorised server-side logic.
- Remove obsolete Google Drive runtime dependencies and unused environment variables only after verifying that no live flow still depends on them.

### Component structure

Organise UI by shared system plus business feature, not one giant UI folder and not one giant page file.

Existing project structure may be adapted gradually. Do not perform a cosmetic folder migration with no functional value.

## 5. Security and data-integrity requirements

### Authentication and access

- Protect all internal routes.
- Redirect unauthenticated users to login.
- Apply least-privilege Row-Level Security to every business table and storage object.
- Never rely only on hidden UI controls for authorisation.
- Admin/service operations must execute only in trusted server code.

### ITR portal credential encryption

Use authenticated reversible encryption because SDDS must retrieve the original portal password.

- Algorithm: AES-256-GCM or an equally reviewed authenticated-encryption implementation.
- AES-256 requires a **32-byte key**. When stored as hexadecimal, that is **64 hex characters**, not 32.
- Keep the key in a server-only environment variable such as `ENCRYPTION_KEY`.
- Never send the encryption key to the browser.
- Never log plaintext passwords, decrypted values, encryption keys, or full ciphertext payloads.
- Store nonce/IV, authentication tag, ciphertext, and encryption version in a defined format.
- Password reveal must be scoped to the selected record; never use one shared eye-toggle state for every client row.

A “missing key” or decryption failure must be treated seriously. If existing data was encrypted with another key, changing the key will not restore it. Do not silently replace ciphertext or pretend recovery succeeded.

### Auditability

Record sensitive and business-critical actions where practical:

- login/security events;
- client creation and key-field changes;
- filing-status changes;
- credential updates or reveals, without recording the plaintext;
- document upload/delete/download actions;
- invoice/payment changes;
- notice/refund status changes;
- backup/export actions.

### Data rules

- Use database constraints for identifiers and critical relationships.
- Avoid destructive cascading deletes for client history.
- Prefer archive/soft-delete for business records where recovery matters.
- Validate data on both client and server boundaries.
- Use transactions for multi-table operations that must succeed together.

## 7. Filing workflow

### Main statuses

Use a controlled workflow rather than free-text status values:

1. New Client
2. Documents Pending
3. Verification Pending
4. Computation In Progress
5. Client Approval Pending
6. Ready To File
7. Filed
8. Completed
9. Rectification Required
10. Notice Received
11. On Hold
12. Cancelled

### Status meaning

- **Filed**: return was submitted; post-filing closure may still be pending.
- **Completed**: required filing, acknowledgement, verification, and applicable closure checks are complete.
- **Rectification Required** and **Notice Received** are active attention states and must not be hidden inside notes.
- **On Hold** requires a reason and next review date where applicable.
- **Cancelled** preserves history and reason; it is not a hard delete.

### Filing records

For each client and assessment year, support multiple linked filing records where applicable:

- Original
- Revised
- Updated
- Belated, where relevant
- Rectification request/response records

Each record should retain its own dates, acknowledgement/reference number, document links, status, notes, and responsible action. Never overwrite the original filing with the revised filing.

## 9. Invoices, payments, and revenue

Revenue management is a core module, not a dashboard decoration.

### Invoice requirements

- Invoice number format: `SDDS/ITR/{AY}/{Serial}`
- Link invoice to client and assessment year.
- Support one or more service line items.
- Store issue date, due date, total, paid amount, balance, status, and notes.
- Default due date: issue date + 30 days; allow an authorised user to edit it.
- Show the due date on the invoice print/PDF.
- Provide a professional printable invoice/PDF.
- Do not assume GST/tax treatment. Keep tax fields configurable until the owner confirms the applicable rule.

### Payment requirements

- Support full and partial payments.
- Store payment date, amount, mode, reference, and note.
- Recalculate balance reliably.
- Prevent paid amount from silently exceeding valid invoice totals without an explicit adjustment/refund flow.

### Invoice statuses

- Draft
- Issued
- Partially Paid
- Paid
- Overdue
- Cancelled

### Revenue reporting

Keep these values distinct:

- invoiced/billed amount;
- collected/received amount;
- outstanding amount;
- overdue amount.

Do not label billed revenue as cash received. Revenue cards must reconcile to invoice and payment records.

Invoice and acknowledgement search must be available where users naturally expect it.

After filing/completion:

- automatically create or enable the next-year follow-up state;
- allow manual exclusion with reason;
- preserve excluded records;
- allow reactivation;
- surface clients due for contact/document collection.

## 11. Design system and exact dashboard direction

### Visual source

For the current redesign:

- Use the approved **blue sidebar layout, colour, and interaction effect** from the designated sidebar reference.
- Use the approved **dashboard content area** from the designated dashboard reference.
- Combine them without creatively reinterpreting the composition.
- Where present in the repository, read:
  - `public/design-reference/sdds-dashboard-target.webp`
  - `docs/dashboard-redesign-spec.md`
  - `docs/dashboard-audit-map.md`
  - `docs/dashboard-component-contract.md`

If these files are absent or conflict with this source, report it; do not guess silently.

### Sidebar active-navigation effect

This requirement is exact and repeatedly misunderstood:

- The sidebar has a blue base/background.
- Inactive navigation items remain on the blue sidebar.
- Exactly one item is active according to the current route.
- The active item receives a spacious white cut-out/tab shape.
- The white shape visually joins or spills into the white dashboard canvas.
- It is an active-navigation treatment, **not** the permanent sidebar background.
- Clicking Clients moves the white effect to Clients and restores Dashboard to the blue inactive state; the same rule applies to every route.
- The active shape must have more breathing space than a tight text highlight.
- Route state, browser refresh, direct URL entry, and nested routes must keep the correct item active.
- Do not remove the effect, replace it with a small pill, or show it on multiple items.

### Visual principles

- Professional accounting/practice-management appearance.
- Clear hierarchy, dense enough to be useful, but not cramped.
- Consistent spacing, radii, shadows, type scale, icon size, and states.
- Use a central token system for colour, typography, spacing, radius, shadow, and transitions.
- Avoid generic AI-dashboard styling: random gradients, glassmorphism, glowing cards, excessive rounded pills, decorative charts, and inconsistent icons.
- Use whitespace deliberately.
- Keep data legible and action-oriented.
- Accessibility: keyboard focus, labels, contrast, semantic controls, and non-colour status cues.

### Responsive behaviour

- Desktop is the primary operating experience.
- No unwanted horizontal page scroll at supported desktop widths.
- Tables may use controlled internal overflow when necessary.
- On smaller screens, collapse the sidebar deliberately; do not merely shrink it until labels break.
- Preserve core actions and status visibility on mobile/tablet.

### UI build order

Do not redesign the whole product by randomly changing one global visual property at a time. Use this order:

1. Audit current shell, routes, components, and visual mismatches.
2. Establish tokens and page grid.
3. Build/fix app shell, sidebar, header, and content canvas.
4. Stabilise typography and shared UI primitives.
5. Rebuild the dashboard sections against the reference.
6. Apply the system to Clients and remaining modules feature by feature.
7. Add responsive states.
8. Run visual, interaction, accessibility, and regression verification.

## 12. Performance standards

The application must not feel delayed after ordinary clicks.

Required practices:

- Navigation feedback must be immediate.
- Avoid blocking route changes on unnecessary client-side fetches.
- Prevent duplicate Supabase queries and fetch waterfalls.
- Keep server and client components intentional.
- Do not make the entire app client-rendered merely for convenience.
- Avoid global state changes that re-render large page trees.
- Paginate or virtualise large data lists where justified.
- Optimise icons, fonts, images, and bundle size.
- Show controlled loading/skeleton states instead of dead clicks.
- Check browser console, network requests, server logs, and Web Vitals before guessing at performance causes.

Never “fix” lag with cosmetic animations that hide slow logic.

## 14. Definition of done

A task is done only when all applicable conditions are true:

### Functional

- The intended workflow works with real project data.
- Direct URL, refresh, back/forward navigation, and nested routes behave correctly.
- Empty, loading, success, validation, and error states exist.

### Visual

- Structure, spacing, dimensions, typography, colours, radii, shadows, icons, alignment, and active states match the approved reference at the baseline viewport.
- Sidebar cut-out behaviour works on every relevant route.
- No obvious overflow, jump, clipping, or inconsistent component styling remains.

### Security/data

- RLS and server boundaries remain valid.
- Sensitive values are not exposed in client code, logs, URLs, or list payloads.
- Existing client, filing, document, invoice, and payment data remain intact.
- Migrations are reversible or backed by a safe migration plan.

### Quality

- Lint, type check, build, and relevant tests pass.
- No new browser-console or server errors.
- No unjustified package, route, schema, or architecture change.
- Documentation is updated when product truth or a permanent contract changes.

## 19. Final decision rule

When forced to choose, prefer the option that:

1. protects client data and credentials;
2. preserves filing and financial history;
3. makes the operator’s next action obvious;
4. matches the approved design accurately;
5. reduces future inconsistency and maintenance;
6. introduces the least unnecessary complexity.

A change that is prettier but less reliable is rejected. A change that is technically clever but harder to operate is questioned. A change that cannot be verified is not declared complete.
